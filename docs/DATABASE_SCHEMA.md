# LeadOS V1 Database Schema Dictionary

LeadOS uses PostgreSQL hosted on Supabase. The schema is highly normalized and optimized with indexing for 500k+ row datasets.

## Core Tables

### `workspaces`
- **Purpose:** Multi-tenant isolation.
- **Fields:** `id`, `name`, `slug`, `config`
- **RLS:** Users can only view workspaces they are members of.

### `workspace_members`
- **Purpose:** Associates users with workspaces.
- **Fields:** `workspace_id`, `user_id`, `role` ('owner', 'admin', 'member')

### `contacts`
- **Purpose:** The core entity representing a person in the network.
- **Fields:** `id`, `name`, `phone`, `roles` (Array), `source`, `origin`
- **Notes:** Uses Soft Delete (`is_deleted = TRUE`). Includes a GIN index on name/phone for fast full-text search. A single contact can have multiple roles (e.g. `opportunity`, `referral_partner`).

### `opportunities`
- **Purpose:** Represents a contact's journey through a recruitment campaign.
- **Fields:** `id`, `contact_id`, `opportunity_type_id`, `status`, `score` (auto-calculated), `next_followup`
- **Notes:** One-to-one relationship with `contacts` currently. Score is computed via trigger.

### `opportunity_types`
- **Purpose:** Configurable recruitment campaigns.
- **Fields:** `id`, `name`, `program_type`, `incentive_amount`
- **Notes:** Defines the pipeline and rules for opportunities.

### `referrals`
- **Purpose:** Network growth tracking.
- **Fields:** `referrer_id` (Contact), `opportunity_id`, `reward_status`, `reward_amount`

### `contact_activities`
- **Purpose:** An event-driven timeline of every action taken on a contact.
- **Fields:** `contact_id`, `activity_type` (Enum), `metadata` (JSON)
- **Notes:** Used for analytics and contact history.

### `daily_snapshots`
- **Purpose:** Historical tracking of a user's performance per day.
- **Fields:** `date`, `leads_collected`, `walk_ins`, `incentives_earned`, `metrics` (JSON)
- **Notes:** Generated by the `rollover_end_day` RPC during the End-Day workflow.

### `sync_queue`
- **Purpose:** Dead-letter queue for offline mutations that persistently fail to sync.
- **Fields:** `entity_type`, `action`, `payload`, `synced`

## Key Stored Procedures (RPCs)

### `calculate_opportunity_score(opportunity_row)`
- **Purpose:** An immutable function that calculates a Readiness Score (0-100) based on interest level, english skills, family support, and proximity. Automatically fired via database trigger on INSERT/UPDATE.

### `check_duplicate_contact(workspace, phone, whatsapp, name)`
- **Purpose:** Fast utility function to detect if a contact already exists in the workspace.

### `rollover_end_day(user_id, workspace_id)`
- **Purpose:** Executes the End-Day workflow securely on the edge. Shifts all tasks with a `next_followup` in the past to tomorrow morning at 9:00 AM.

## Indexes
Heavy indexing is applied to ensure Kanban and Timeline queries remain instantaneous:
- `idx_contacts_workspace` (Partial index where `is_deleted = FALSE`)
- `idx_opportunities_status`
- `idx_contacts_search` (GIN Full Text)
- `idx_activities_contact` (Ordered by `created_at DESC`)
